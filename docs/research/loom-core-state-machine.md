Architecture Overview\n\n### Project Structure\n\nLoom is a Cargo workspace with the following key crates:\n\n- **loom-core** — Agent state machine, core abstractions, type definitions\n- **loom-server** — HTTP API server with server-side LLM proxy\n- **loom-cli** — Command-line interface\n- **loom-thread** — Conversation persistence and sync (thread state, message history)\n- **loom-llm-\\*** — LLM provider integrations (Anthropic, OpenAI)\n- **loom-tools** — Agent tool registry and execution framework\n- **loom-auth\\*** — Authentication and authorization\n- **loom-git** — VCS integration (likely for PostToolsHook auto-commit)\n- **loom-auto-commit** — Automation for commit generation\n\n**Web Frontend:** SvelteKit + Tailwind, backed by SQLite database.\n\n### Design Principles\n\n1. **Modularity** — Clean separation between core abstractions, LLM providers, tools\n2. **Extensibility** — Trait-based implementations for new providers and tools\n3. **Reliability** — Robust error handling with retry mechanisms and structured logging\n\n### Server-Side Proxy Architecture\n\nThe system uses a **server-side proxy model** where:\n- **CLI** makes HTTP/SSE requests to **loom-server**\n- **loom-server** acts as a proxy between CLI and LLM providers (Anthropic, OpenAI)\n- API credentials remain server-side only\n- Clients communicate via HTTP and Server-Sent Events (SSE) for streaming\n\nThis design ensures security and rate-limit management at the server layer.\n\n---\n\n## Part II: The Core Agent State Machine\n\n### States\n\nThe agent state machine has **seven core states**:\n\n#### 1. **WaitingForUserInput**\n- **Purpose:** Idle state awaiting user action or tool results to process\n- **Entry Condition:** Agent initialization, completion of previous cycle\n- **Responsibilities:**\n  - Display prompt or conversation context to user\n  - Await input (new prompt, feedback, approval for tool execution)\n  - Collect user input or proceed with cached results from prior tool execution\n- **Transition Trigger:** User provides input or tool results are ready to process\n\n#### 2. **CallingLlm**\n- **Purpose:** Request inference from the LLM\n- **Entry Condition:** User input received or tool results ready to send for further processing\n- **Responsibilities:**\n  - Build inference request with conversation history, system prompt, tools schema\n  - Add user input (or tool results) to message history\n  - Call LLM via server-side proxy (async over HTTP/SSE)\n  - Stream or collect LLM response\n- **Exit Condition:** LLM response received (may include multiple content blocks, tool calls)\n- **Transition Trigger:** LlmResponseReceived event\n\n#### 3. **ProcessingLlmResponse**\n- **Purpose:** Parse and validate the LLM response\n- **Entry Condition:** LlmResponseReceived event (response body available)\n- **Responsibilities:**\n  - Parse response JSON/content blocks\n  - Identify response type:\n    - **Text-only response** → display to user (shortcut to WaitingForUserInput or ShuttingDown)\n    - **Tool use request(s)** → extract tool names, parameters\n  - Validate tool schema against known tools (catch hallucinated tools)\n  - Extract stop/finish reason (e.g., `stop_reason: \"end_turn\"`)\n  - Build list of ToolCall objects (name, input, ID)\n- **Transition Trigger:**\n  - No tools requested → DisplayMessage action, return to WaitingForUserInput\n  - Tools requested → ExecuteTools action, move to ExecutingTools\n  - LLM signals end-of-turn → ShuttingDown (if top-level goal complete)\n\n#### 4. **ExecutingTools**\n- **Purpose:** Run agent-requested tools and collect results\n- **Entry Condition:** ProcessingLlmResponse identified tool calls\n- **Responsibilities:**\n  - For each tool in request:\n    - Look up tool handler (via tool registry in loom-tools)\n    - Execute handler with input parameters\n    - Capture output (stdout, return value) or error (stderr, exception)\n    - Build ToolResult object (tool_name, output/error, execution_time)\n  - Collect all results into a single ToolExecutionCompleted event payload\n  - On error: capture error context (retryable? fatal?)\n- **Transition Trigger:** ToolExecutionCompleted event (all tools done or fatal error)\n\n#### 5. **PostToolsHook**\n- **Purpose:** Run post-execution automation (auto-commit, analytics, cleanup)\n- **Entry Condition:** ToolExecutionCompleted event\n- **Responsibilities:**\n  - **Auto-commit (via loom-auto-commit):**\n    - Detect file changes (git status)\n    - Generate commit message using Haiku LLM (small, fast model)\n    - Commit changes with `git add -A && git commit -m \"...\"`\n  - **Analytics & Logging:**\n    - Record tool execution metrics (duration, success/failure, tokens used)\n    - Store interaction in audit log\n  - **Cleanup:**\n    - Persist conversation state (thread) to database\n    - Update thread metadata (timestamps, summary)\n  - **Conditional Flow:**\n    - If errors in tool execution → may loop back to CallingLlm or Error state\n    - If success → loop back to WaitingForUserInput or CallingLlm (for next turn)\n- **Transition Trigger:** PostToolsHookCompleted event\n\n#### 6. **Error**\n- **Purpose:** Handle failures with retry and recovery logic\n- **Entry Condition:** Exception in CallingLlm, ExecutingTools, or PostToolsHook\n- **Responsibilities:**\n  - Classify error type:\n    - **Transient** (rate limit, timeout, network) → retry\n    - **Fatal** (invalid input, auth failure, unsupported tool) → escalate to user\n  - **Retry Logic:**\n    - Use **exponential backoff**: wait = base_delay × (backoff_multiplier ^ attempt_count)\n    - Common settings: base_delay=1s, multiplier=2, max_attempts=3–5\n    - After max retries exceeded → escalate to user\n  - **Error Display:**\n    - Construct user-friendly error message (hide internal details unless debug mode)\n    - Suggest recovery action (retry, try different input, check permissions)\n  - **Logging:**\n    - Log full error context (stack trace, input, state) for debugging\n- **Transition Trigger:**\n  - Retry remaining → back to CallingLlm or ExecutingTools\n  - Max retries exceeded → DisplayMessage (to user), then ShuttingDown or WaitingForUserInput\n  - Fatal error → ShuttingDown with error code\n\n#### 7. **ShuttingDown**\n- **Purpose:** Clean termination of agent session\n- **Entry Condition:**\n  - User request (e.g., `exit`, `quit` command)\n  - Conversation complete (LLM signals `stop_reason: \"end_turn\"` or goal achieved)\n  - Fatal error with no recovery\n- **Responsibilities:**\n  - Final cleanup:\n    - Flush any pending writes (thread state, logs)\n    - Close LLM connection\n    - Persist final conversation state to database\n  - **Exit code:**\n    - 0 = success\n    - 1 = error\n  - Display closing message to user (if interactive)\n- **Terminal State:** No further transitions; process exits\n\n---\n\n## Part III: Events\n\n### Event Types\n\nEach event encapsulates data that triggers a state transition. Events are immutable payloads passed to the state machine's handler.\n\n#### 1. **UserInputReceived**\n- **Triggered By:** User types input in WaitingForUserInput state\n- **Payload:**\n  - `input: String` — user text (prompt, feedback, tool approval)\n  - `timestamp: Instant` — when input was received\n  - `context: ConversationContext` — reference to current thread\n- **Triggers Transition:** WaitingForUserInput → CallingLlm\n\n#### 2. **LlmResponseReceived**\n- **Triggered By:** LLM returns inference result (after HTTP/SSE stream completes)\n- **Payload:**\n  - `response: LlmResponse` — structured response (see Conversation Model below)\n  - `tokens_used: TokenCount` — input + output tokens\n  - `latency_ms: u64` — round-trip inference time\n  - `model: String` — LLM model name (e.g., \"claude-3-5-sonnet\")\n- **Triggers Transition:** CallingLlm → ProcessingLlmResponse\n\n#### 3. **ToolsRequested** (internal event during ProcessingLlmResponse)\n- **Triggered By:** LLM response contains tool_use blocks\n- **Payload:**\n  - `tools: Vec<ToolCall>` — list of tools to execute (each with name, input, ID)\n- **Triggers Transition:** ProcessingLlmResponse → ExecutingTools\n\n#### 4. **ToolExecutionCompleted**\n- **Triggered By:** All requested tools have finished (success or error)\n- **Payload:**\n  - `results: Vec<ToolResult>` — results for each tool\n    - `tool_name: String`\n    - `output: String` (success) or `error: String` (failure)\n    - `execution_time_ms: u64`\n  - `total_duration_ms: u64`\n  - `any_errors: bool` — quick flag for error checking\n- **Triggers Transition:** ExecutingTools → PostToolsHook (or Error if fatal)\n\n#### 5. **PostToolsHookCompleted**\n- **Triggered By:** Auto-commit, logging, and cleanup tasks finish\n- **Payload:**\n  - `commit_hash: Option<String>` — git commit SHA if auto-commit occurred\n  - `hook_duration_ms: u64`\n  - `success: bool`\n- **Triggers Transition:** PostToolsHook → WaitingForUserInput or CallingLlm (for next turn)\n\n#### 6. **ErrorOccurred**\n- **Triggered By:** Exception or failure in any state\n- **Payload:**\n  - `error: AgentError` — error type and message\n  - `state: AgentState` — which state failed\n  - `retryable: bool` — can we retry?\n  - `attempt_count: u32` — which retry attempt (0 = first)\n  - `context: String` — additional context for debugging\n- **Triggers Transition:** (any state) → Error\n\n#### 7. **ShutdownRequested**\n- **Triggered By:** User command, timeout, or fatal error\n- **Payload:**\n  - `reason: ShutdownReason` — why shutdown? (UserRequest, Timeout, FatalError, GoalComplete)\n  - `exit_code: i32` — desired process exit code\n  - `message: Option<String>` — message to display to user\n- **Triggers Transition:** (any state) → ShuttingDown\n\n#### 8. **RetryTriggered**\n- **Triggered By:** Error state after exponential backoff delay\n- **Payload:**\n  - `attempt_number: u32` — current retry count\n  - `next_delay_ms: u64` — time waited before retry\n  - `original_error: AgentError` — the error being retried\n- **Triggers Transition:** Error → CallingLlm or ExecutingTools (depending on original failure)\n\n---\n\n## Part IV: Actions\n\n### Action Type System\n\nWhen `handle_event(state, event)` is called, the state machine returns an `Action` that tells the **caller** (the event loop) what to do next. This keeps the state machine **pure** (no I/O side effects).\n\n#### Action Enum\n\n```rust\npub enum Action {\n    SendLlmRequest(LlmRequest),\n    ExecuteTools(Vec<ToolCall>),\n    DisplayMessage(String),\n    RunPostToolsHook(PostToolsHookContext),\n    RetryWithBackoff { delay_ms: u64, next_event: Box<Event> },\n    Shutdown(ShutdownContext),\n    WaitForInput(WaitContext),\n    Continue, // No-op; ready for next event\n}\n```\n\n#### Action Details\n\n##### 1. **SendLlmRequest(LlmRequest)**\n- **When:** After UserInputReceived or ToolExecutionCompleted in CallingLlm state\n- **Payload:**\n  - `messages: Vec<Message>` — full conversation history\n  - `system: String` — system prompt\n  - `tools: Vec<ToolDefinition>` — available tools schema\n  - `max_tokens: u32`\n  - `temperature: f32`\n  - `model: String` — which LLM to call\n- **Caller Responsibility:**\n  - Make HTTP POST to loom-server /proxy/anthropic/complete or /proxy/openai/complete\n  - Stream response via SSE\n  - On completion, emit LlmResponseReceived event\n  - On error, emit ErrorOccurred event\n\n##### 2. **ExecuteTools(Vec<ToolCall>)**\n- **When:** After ProcessingLlmResponse identifies tool requests\n- **Payload:**\n  - `tools: Vec<ToolCall>` — list of tools to invoke\n    - `id: String` — LLM's tool ID (to match result back)\n    - `name: String` — tool name\n    - `input: serde_json::Value` — tool parameters\n- **Caller Responsibility:**\n  - Look up each tool in registry (loom-tools)\n  - Execute tool handler (may be local subprocess, RPC, or HTTP)\n  - Capture output/error for each\n  - Emit ToolExecutionCompleted event with results\n  - On execution error, emit ErrorOccurred event\n\n##### 3. **DisplayMessage(String)**\n- **When:** LLM response is text-only (no tool calls)\n- **Payload:**\n  - `message: String` — response text to show user\n- **Caller Responsibility:**\n  - Print to stdout or display in UI\n  - Transition back to WaitingForInput\n\n##### 4. **RunPostToolsHook(PostToolsHookContext)**\n- **When:** After ToolExecutionCompleted in PostToolsHook state\n- **Payload:**\n  - `changed_files: Vec<PathBuf>` — files modified by tools\n  - `thread_id: String` — conversation thread identifier\n  - `use_haiku_for_commit: bool` — whether to auto-commit\n  - `git_repo_path: PathBuf` — path to git repo\n  - `commit_scope: Option<String>` — e.g., \"feat\", \"fix\" (conventional commits)\n- **Caller Responsibility:**\n  - Detect modified files via `git status`\n  - If `use_haiku_for_commit`, call Haiku LLM to generate commit message\n    - Prompt: \"Summarize these code changes in 1 line (conventional format): ...\"\n    - Use response as commit message\n  - Run `git add -A && git commit -m \"...\"`\n  - Persist thread state to database (via loom-thread)\n  - Emit PostToolsHookCompleted event with commit hash\n  - On error, emit ErrorOccurred event\n\n##### 5. **RetryWithBackoff { delay_ms: u64, next_event: Box<Event> )**\n- **When:** Error state with retryable error and attempts remaining\n- **Payload:**\n  - `delay_ms: u64` — milliseconds to wait before retry\n    - Formula: `base_delay_ms × (backoff_multiplier ^ attempt)`\n    - Example: base=500ms, multiplier=2, attempt=2 → 500 × 4 = 2000ms\n  - `next_event: Box<Event>` — the event to emit after delay (e.g., UserInputReceived, ToolExecutionCompleted)\n- **Caller Responsibility:**\n  - Sleep for `delay_ms`\n  - Emit `next_event` to state machine\n  - Track attempt count (incremented in Error state)\n\n##### 6. **Shutdown(ShutdownContext)**\n- **When:** ShuttingDown state after cleanup\n- **Payload:**\n  - `exit_code: i32` — process exit code (0 or 1)\n  - `message: String` — final message to user\n- **Caller Responsibility:**\n  - Print message to user\n  - Exit process with given exit code\n\n##### 7. **WaitForInput(WaitContext)**\n- **When:** WaitingForUserInput state (ready for next input)\n- **Payload:**\n  - `prompt: String` — what to prompt user for (e.g., \"Next command:\")\n  - `timeout_ms: Option<u64>` — optional timeout to auto-shutdown\n- **Caller Responsibility:**\n  - Display prompt\n  - Read input from stdin or UI\n  - Emit UserInputReceived event\n  - If timeout, emit ShutdownRequested event\n\n##### 8. **Continue**\n- **When:** No action needed; ready for next event\n- **Payload:** None\n- **Caller Responsibility:**\n  - Proceed to next event in queue\n\n---\n\n## Part V: Pure State Machine Design\n\n### The Core Principle: Functional State Transitions\n\nLoom's state machine is **pure functional**, meaning:\n\n```rust\nfn handle_event(state: AgentState, event: Event, context: &AgentContext) -> Action\n```\n\n**Key Properties:**\n\n1. **No Internal I/O** — State machine does not perform HTTP calls, file I/O, subprocess execution, etc.\n2. **Deterministic** — Same (state, event, context) always produces same Action\n3. **Immutable State** — AgentState is read-only; handler returns Action describing next state\n4. **Separation of Concerns:**\n   - **State Machine Logic** (in loom-core): decide what to do\n   - **I/O & Side Effects** (in caller event loop): actually do it\n\n### Event Loop Pattern\n\n```\nloop {\n    state = initial_state\n    loop {\n        event = wait_for_event(state, timeout)\n        action = handle_event(state, event, &context)\n        state = perform_action(action)  // executes I/O, returns new state\n        if is_terminal(state) { break }\n    }\n}\n```\n\n### Benefits\n\n- **Testability:** State machine logic is testable without mocking I/O\n- **Reproducibility:** Can replay event sequences to debug\n- **Composability:** Easy to extend with new states/events without breaking core logic\n- **Formal Verification:** Pure state machines can be model-checked\n\n---\n\n## Part VI: Conversation Model\n\n### Message Type Hierarchy\n\n#### `Message`\nRepresents a single turn in the conversation history.\n\n```\nMessage {\n    role: MessageRole,  // \"user\" | \"assistant\" | \"tool\"\n    content: Vec<ContentBlock>,\n    timestamp: Instant,\n    metadata: Map<String, Value>,\n}\n```\n\n#### `MessageRole`\n- **User** — Input from human\n- **Assistant** — Response from LLM\n- **Tool** — Result from tool execution\n\n#### `ContentBlock` (Enum)\n```\nContentBlock::Text(TextBlock) {\n    text: String,\n}\n\nContentBlock::ToolUse(ToolUseBlock) {\n    id: String,           // LLM-assigned ID\n    name: String,         // Tool name\n    input: serde_json::Value,\n}\n\nContentBlock::ToolResult(ToolResultBlock) {\n    tool_use_id: String,  // References the ToolUseBlock.id\n    content: String,      // Output or error message\n    is_error: bool,\n}\n```\n\n### Conversation History Example\n\n```\nMessage {\n    role: User,\n    content: [TextBlock(\"Create a function to sort an array\")]\n}\n\nMessage {\n    role: Assistant,\n    content: [\n        TextBlock(\"I'll write a sorting function for you:\"),\n        ToolUse { id: \"tool_0\", name: \"WriteFile\", input: { \"path\": \"sort.rs\", \"content\": \"...\" } }\n    ]\n}\n\nMessage {\n    role: Tool,\n    content: [ToolResult { tool_use_id: \"tool_0\", content: \"File written\", is_error: false }]\n}\n\nMessage {\n    role: Assistant,\n    content: [TextBlock(\"Done! I've created sort.rs with a quicksort implementation.\")]\n}\n```\n\n### LLM Request Type\n\n```\nLlmRequest {\n    messages: Vec<Message>,     // Full history\n    system: String,             // System prompt\n    tools: Vec<ToolDefinition>, // Available tools\n    max_tokens: u32,\n    temperature: f32,\n    model: String,\n    stream: bool,               // Use SSE streaming?\n}\n```\n\n### LLM Response Type\n\n```\nLlmResponse {\n    content: Vec<ContentBlock>,  // Usually one TextBlock or ToolUseBlock\n    stop_reason: String,         // \"end_turn\", \"tool_use\", \"max_tokens\"\n    usage: TokenUsage {\n        input_tokens: u32,\n        output_tokens: u32,\n    },\n    model: String,\n    latency_ms: u64,\n}\n```\n\n### Tool Definition Schema\n\n```\nToolDefinition {\n    name: String,\n    description: String,\n    input_schema: JsonSchema,  // JSON Schema describing parameters\n}\n\nExample:\n{\n    \"name\": \"WriteFile\",\n    \"description\": \"Write content to a file\",\n    \"input_schema\": {\n        \"type\": \"object\",\n        \"properties\": {\n            \"path\": { \"type\": \"string\", \"description\": \"File path\" },\n            \"content\": { \"type\": \"string\", \"description\": \"File content\" }\n        },\n        \"required\": [\"path\", \"content\"]\n    }\n}\n```\n\n---\n\n## Part VII: PostToolsHook & Auto-Commit\n\n### Flow\n\nAfter tools finish execution (ToolExecutionCompleted event), the state machine enters **PostToolsHook** state:\n\n1. **Detect Changes:**\n   - Run `git status --porcelain` to see modified, staged, untracked files\n   - Filter to relevant files (exclude build artifacts, temp files)\n\n2. **Generate Commit Message (via Haiku):**\n   - Call Haiku LLM (small, fast model)\n   - Prompt:\n     ```\n     You are a code commit message generator. Summarize these changes in conventional format (type: scope: message).\n\n     Changed files:\n     {list of modified paths}\n\n     Diffs:\n     {git diff output}\n\n     Commit message (1 line):\n     ```\n   - Receive 1-line message (e.g., \"feat: add sorting utility\")\n\n3. **Commit:**\n   - Run `git add -A`\n   - Run `git commit -m \"feat: add sorting utility\"`\n   - Capture commit SHA\n\n4. **Persist Thread State:**\n   - Save conversation state to database (via loom-thread)\n   - Update thread metadata (last_modified, summary)\n\n5. **Emit PostToolsHookCompleted:**\n   - Return Action with commit hash\n   - Loop back to WaitingForUserInput or CallingLlm for next turn\n\n### AgentConfig Parameters\n\nThe behavior of PostToolsHook is controlled by **AgentConfig**:\n\n```\nAgentConfig {\n    // LLM Settings\n    llm_model: String,                    // \"claude-3-5-sonnet\"\n    llm_temperature: f32,                 // 0.0–1.0\n    max_tokens: u32,                      // 4096\n\n    // Tool Settings\n    available_tools: Vec<String>,         // Which tools to register\n    timeout_per_tool_ms: u64,             // How long to wait for each tool\n\n    // PostToolsHook Settings\n    auto_commit_enabled: bool,            // Enable auto-commit?\n    haiku_for_commits: bool,              // Use Haiku for message generation?\n    commit_scope: Option<String>,         // \"feat\", \"fix\", etc.\n\n    // Error Handling\n    max_retries: u32,                     // Max retry attempts\n    initial_backoff_ms: u64,              // Base delay for retries\n    backoff_multiplier: f32,              // Exponential backoff factor (e.g., 2.0)\n\n    // Thread Persistence\n    thread_db_path: PathBuf,              // Where to store conversations\n\n    // Behavior\n    stream_responses: bool,               // Use SSE streaming?\n    debug_logging: bool,                  // Verbose logs?\n}\n```\n\n---\n\n## Part VIII: Error Handling & Retry Logic\n\n### Error State Behavior\n\nWhen an error occurs, the state machine enters the **Error** state:\n\n```rust\nenum ErrorContext {\n    retryable: bool,\n    attempt_count: u32,\n    max_attempts: u32,\n    backoff_multiplier: f32,\n    original_error: AgentError,\n    state_when_failed: AgentState,\n}\n```\n\n### Exponential Backoff Algorithm\n\n```\nif error.retryable && attempt < max_attempts {\n    delay_ms = initial_backoff_ms × (backoff_multiplier ^ attempt)\n    // Example: 500 × (2.0 ^ 2) = 2000ms for attempt 2\n\n    return Action::RetryWithBackoff {\n        delay_ms,\n        next_event: retry_event,\n    }\n} else {\n    return Action::Shutdown(error_message)\n}\n```\n\n### Retryable vs. Fatal Errors\n\n**Retryable:**\n- Network timeouts\n- Rate limits (HTTP 429)\n- Transient provider outages (HTTP 5xx)\n- Tool execution timeouts\n\n**Fatal:**\n- Invalid API key (HTTP 401)\n- Malformed request (HTTP 400)\n- Unknown tool name\n- Unsupported model\n\n---\n\n## Part IX: false\n```\n\n---\n\n## Part X: Testing Strategy\n\n### Property-Based Testing\n\nLoom uses **property-based tests** (via proptest or quickcheck) to verify state machine correctness:\n\n#### Example Properties\n\n1. **Idempotence:** Replaying the same event sequence always produces the same final state\n2. **Completeness:** Every state-event pair is defined (no panic conditions)\n3. **Termination:** Agent always reaches ShuttingDown state (no infinite loops)\n4. **Message Integrity:** Conversation history is never corrupted (messages are immutable)\n\n### Test Coverage\n\nFrom verification.md:\n- **Authorization test suite:** 193 tests (100% pass)\n- **Branch protection enforcement:** 19 unit tests + 8 property-based tests\n- **State machine:** Comprehensive transition table tests\n\n### Shrinking & Fuzzing\n\nProperty-based testing frameworks automatically shrink failing counterexamples:\n- If a 10-event sequence causes failure, the framework tries 9-event subsequences\n- Shrinks to minimal reproducer for debugging\n\n---\n\n## Part XI: Threading & Persistence\n\n### Thread System (loom-thread crate)\n\n**Thread = Conversation Session**\n\nEach thread is identified by a unique ID and contains:\n- `thread_id: String` — UUID or similar\n- `messages: Vec<Message>` — full conversation history (immutable append-only)\n- `metadata: Map<String, Value>` — custom metadata\n- `created_at, updated_at: Timestamp`\n- `status: ThreadStatus` — active, archived, error\n\n### Persistence Model\n\n- Threads are stored in SQLite database\n- Each message append is persisted atomically\n- Supports search and retrieval of past conversations\n- Enables resumption of interrupted sessions\n\n---\n\n## Part XII: Specification Documents\n\nLoom's design is documented Adopt\n\n1. **Pure State Machine:** Keep state transitions free of I/O; return Actions for the caller to execute\n2. **Immutable Events:** Events are immutable payloads; no internal state
